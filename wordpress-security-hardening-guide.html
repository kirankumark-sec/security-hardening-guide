01
Restrict wp-login.php & wp-admin by IP
CRITICAL Server Level ApacheNginx.htaccess
⌄
⚠️ Issue
WordPress admin login page and dashboard are publicly accessible, exposing them to automated brute-force, credential-stuffing, and targeted attacks.
🔥 Risk
Unauthorized admin access, site takeover, data exfiltration, ransomware deployment.
📁 Files & Paths
/var/www/html/.htaccess/etc/apache2/sites-enabled/your-site.conf/etc/nginx/sites-enabled/your-site.conf
🔧 Fix — Apache (.htaccess or VirtualHost)
/var/www/html/.htaccess
# ─── Restrict wp-login.php ───────────────────────────────────
<Files wp-login.php>
    Order deny,allow
    Deny from all
    Allow from 203.0.113.10
    Allow from 203.0.113.20
</Files>

# ─── Restrict wp-admin directory ─────────────────────────────
<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteCond %{REQUEST_URI} ^/wp-admin
    RewriteCond %{REMOTE_ADDR} !^203\.0\.113\.10$
    RewriteCond %{REMOTE_ADDR} !^203\.0\.113\.20$
    RewriteRule ^ - [F,L]
</IfModule>

# ─── Allow admin-ajax.php (required by plugins) ──────────────
<Files admin-ajax.php>
    Order allow,deny
    Allow from all
</Files>
🔧 Fix — Nginx (server block)
/etc/nginx/sites-enabled/your-site.conf
location = /wp-login.php {
    allow 203.0.113.10;
    allow 203.0.113.20;
    deny  all;
    fastcgi_pass unix:/run/php/php8.2-fpm.sock;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}

location ~ ^/wp-admin/ {
    allow 203.0.113.10;
    allow 203.0.113.20;
    deny  all;
    try_files $uri $uri/ /index.php?$args;
    fastcgi_pass unix:/run/php/php8.2-fpm.sock;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}

location = /wp-admin/admin-ajax.php {
    allow all;
    fastcgi_pass unix:/run/php/php8.2-fpm.sock;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}
👥 Responsible
Web Server Admin Linux/System Admin
🔄 Restart
⟳ Apache: systemctl reload apache2
⟳ Nginx: systemctl reload nginx
✅ Verification
$ curl -I https://your-site.com/wp-login.php → HTTP/2 403 $ curl -I https://your-site.com/wp-admin/ → HTTP/2 403 $ curl -I https://your-site.com/wp-admin/admin-ajax.php → HTTP/2 200 or 400 (not 403)
📋 VAPT Closure Statement
🔒
Audit Closure
Access to wp-login.php and /wp-admin/ has been restricted at the web server layer via IP allowlisting, returning HTTP 403 for all unauthenticated source addresses. admin-ajax.php availability for plugin functionality is preserved.
02
Disable XML-RPC Securely
CRITICALServer + WordPressxmlrpc.php.htaccessfunctions.php
⌄
⚠️ Issue
XML-RPC is enabled by default, allowing remote method invocations. Attackers abuse system.multicall to perform thousands of login attempts in a single HTTP request.
🔥 Risk
Credential bruteforce amplification (1 request = 500 guesses), DDoS amplification, unauthorized remote content publication.
🔧 Fix — Apache
/var/www/html/.htaccess
<Files xmlrpc.php>
    Order deny,allow
    Deny from all
</Files>
🔧 Fix — Nginx
/etc/nginx/sites-enabled/your-site.conf
location = /xmlrpc.php {
    deny all;
    access_log off;
    log_not_found off;
    return 444;
}
🔧 Fix — WordPress layer
functions.php
add_filter( 'xmlrpc_enabled', '__return_false' );
add_filter( 'wp_headers', function( $h ) { unset( $h['X-Pingback'] ); return $h; } );
add_filter( 'xmlrpc_methods', function( $m ) { unset( $m['pingback.ping'] ); return $m; } );
👥 Responsible
Web Server Admin WordPress Developer
🔄 Restart
⟳ systemctl reload apache2 / nginx
✅ Verification
$ curl -I https://your-site.com/xmlrpc.php → HTTP/2 403 (Apache) or connection refused (Nginx 444) $ curl -I https://your-site.com/ | grep -i pingback → (empty)
📋 VAPT Closure Statement
🔒
Audit Closure
XML-RPC endpoint disabled at web server (HTTP 403/444) and WordPress layer via xmlrpc_enabled filter. X-Pingback header suppressed. Endpoint inaccessible to external requests.
03
Brute-Force Protection & Login Rate Limiting
CRITICALServer + Pluginfail2banNginxWP Plugin
⌄
⚠️ Issue
WordPress imposes no native limit on authentication attempts, making it susceptible to automated credential-stuffing and password-spray attacks.
🔥 Risk
Account takeover via dictionary/brute-force attacks, server resource exhaustion, DoS conditions.
🔧 Fix 1 — Nginx Rate Limiting
/etc/nginx/nginx.conf (http block)
limit_req_zone $binary_remote_addr zone=wp_login:10m rate=10r/m;
/etc/nginx/sites-enabled/your-site.conf
location = /wp-login.php {
    limit_req zone=wp_login burst=5 nodelay;
    limit_req_status 429;
    allow 203.0.113.10;
    deny  all;
    fastcgi_pass unix:/run/php/php8.2-fpm.sock;
    include fastcgi_params;
    fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
}
🔧 Fix 2 — Fail2Ban
/etc/fail2ban/filter.d/wordpress.conf
[Definition]
failregex = ^<HOST> .* "POST /wp-login.php
            ^<HOST> .* "POST /xmlrpc.php
ignoreregex =
/etc/fail2ban/jail.local
[wordpress-login]
enabled   = true
filter    = wordpress
logpath   = /var/log/nginx/access.log
maxretry  = 5
findtime  = 300
bantime   = 3600
action    = iptables-multiport[name=wordpress, port="80,443"]
👥 Responsible
Linux/System Admin Web Server Admin
🔄 Restart
⟳ systemctl restart fail2ban
⟳ systemctl reload nginx
✅ Verification
$ for i in {1..15}; do curl -s -o /dev/null -w "%{http_code}\n" -X POST https://your-site.com/wp-login.php; done → First 10: 200/302, then: 429 $ fail2ban-client status wordpress-login → Shows banned IPs
📋 VAPT Closure Statement
🔒
Audit Closure
Brute-force protection via Nginx rate limiting (10 req/min, burst 5, HTTP 429) backed by Fail2Ban (1-hour IP ban after 5 failed attempts in 5 minutes). All controls active and verified.
04
Enable & Enforce Two-Factor Authentication (2FA)
HIGHWordPress PluginWP Two Factorfunctions.php
⌄
⚠️ Issue
WordPress uses single-factor authentication by default. Compromised credentials alone are sufficient for admin access.
🔥 Risk
Full admin compromise via stolen, leaked, or phished credentials. Password policies alone are insufficient.
🔧 Fix — Install & Configure Two Factor Plugin
WP CLI Installation
$ wp plugin install two-factor --activate --path=/var/www/html
🔧 Fix — Force 2FA for Admin roles (functions.php)
functions.php
add_action( 'init', function() {
    if ( ! is_user_logged_in() ) return;
    $user = wp_get_current_user();
    $required_roles = [ 'administrator', 'editor' ];
    if ( array_intersect( $required_roles, (array) $user->roles ) ) {
        if ( class_exists( 'Two_Factor_Core' ) ) {
            $providers = Two_Factor_Core::get_enabled_providers_for_user( $user );
            if ( empty( $providers ) ) {
                if ( ! in_array( $GLOBALS['pagenow'], ['profile.php', 'user-edit.php'] ) ) {
                    wp_redirect( admin_url( 'profile.php#two-factor-options' ) );
                    exit;
                }
            }
        }
    }
} );
👥 Responsible
WordPress Developer Linux/System Admin
🔌 Supported Methods
  • TOTP (Google Authenticator, Authy)
  • FIDO2/WebAuthn (hardware keys)
  • Email-based OTP
  • Backup verification codes
✅ Verification
$ wp user meta get <user_id> _two_factor_enabled_providers --path=/var/www/html → Should list configured providers (totp, fido-u2f, etc.)
📋 VAPT Closure Statement
🔒
Audit Closure
MFA (TOTP-based) enforced for all Administrator and Editor accounts. Users without 2FA enrollment are programmatically redirected to configure a second factor before accessing admin functionality.
05
Strong Password Policy Enforcement
HIGHWordPress Core + Pluginfunctions.php
⌄
⚠️ Issue
WordPress allows weak passwords — admins can bypass the "Weak Password" warning. No minimum complexity or length enforcement exists by default.
🔥 Risk
Accounts with weak passwords are vulnerable to dictionary and credential-stuffing attacks.
🔧 Fix — Custom Password Validation
functions.php
add_action( 'user_profile_update_errors', 'enforce_strong_password', 10, 3 );
add_action( 'validate_password_reset', 'enforce_strong_password', 10, 2 );

function enforce_strong_password( $errors, $update = null, $user = null ) {
    $pass1 = isset( $_POST['pass1'] ) ? trim( $_POST['pass1'] ) : '';
    if ( empty( $pass1 ) ) return;
    $e = [];
    if ( strlen( $pass1 ) < 12 )            $e[] = 'At least 12 characters';
    if ( ! preg_match( '/[A-Z]/', $pass1 ) ) $e[] = 'One uppercase letter';
    if ( ! preg_match( '/[a-z]/', $pass1 ) ) $e[] = 'One lowercase letter';
    if ( ! preg_match( '/[0-9]/', $pass1 ) ) $e[] = 'One number';
    if ( ! preg_match( '/[\W_]/', $pass1 ) )  $e[] = 'One special character';
    if ( ! empty( $e ) ) {
        $errors->add( 'weak_password', 'Password must include: ' . implode( ', ', $e ) . '.' );
    }
}
👥 Responsible
WordPress Developer
📏 Policy
  • Min 12 characters
  • Uppercase + lowercase
  • At least 1 number
  • At least 1 special character
  • No weak-password bypass
📋 VAPT Closure Statement
🔒
Audit Closure
Password complexity policy (min 12 chars, uppercase, lowercase, numeric, special) enforced at WordPress application layer on profile updates and password resets. Weak-password bypass disabled.
06
Prevent User Enumeration via REST API & Author URLs
MEDIUMWordPressfunctions.php.htaccess
⌄
⚠️ Issue
WordPress exposes usernames via /?author=1, REST API /wp-json/wp/v2/users, and author archive pages.
🔥 Risk
Valid username disclosure reduces brute-force effort by 50%. Combined with leaked password databases it enables targeted credential attacks.
🔧 Fix 1 — Block author=N enumeration
.htaccess (Apache)
<IfModule mod_rewrite.c>
    RewriteEngine On
    RewriteCond %{QUERY_STRING} author=\d
    RewriteRule ^ /? [L,R=301]
</IfModule>
🔧 Fix 2 — Restrict REST API users endpoint
functions.php
add_filter( 'rest_endpoints', function( $endpoints ) {
    if ( isset( $endpoints['/wp/v2/users'] ) ) {
        foreach ( $endpoints['/wp/v2/users'] as $key => $ep ) {
            $endpoints['/wp/v2/users'][$key]['permission_callback'] = function() {
                return current_user_can( 'list_users' );
            };
        }
    }
    return $endpoints;
} );

add_action( 'template_redirect', function() {
    if ( is_author() ) { wp_redirect( home_url( '/' ), 301 ); exit; }
} );
👥 Responsible
WordPress Developer Web Server Admin
✅ Verification
$ curl -I "https://your-site.com/?author=1" → 403 or 301 $ curl -s https://your-site.com/wp-json/wp/v2/users → rest_forbidden
📋 VAPT Closure Statement
🔒
Audit Closure
Username enumeration vectors mitigated: ?author=N blocked at web server layer, /wp-json/wp/v2/users requires authentication (HTTP 401 for unauthenticated), author archives redirect to homepage.

remove_action( 'wp_head', 'wp_generator' );

function remove_wp_version_strings( $src ) {
    if ( strpos( $src, 'ver=' ) !== false ) { $src = remove_query_arg( 'ver', $src ); }
    return $src;
}
Shell — delete disclosure files

$ rm /var/www/html/license.txt
👥 Responsible
WordPress Developer Linux/System Admin

File Upload Hardening
HIGHServer + WordPress.htaccessnginx.conffunctions.php
⌄
⚠️ Issue
The WordPress upload directory allows file uploads. If PHP execution is not disabled, attackers can upload webshells disguised as images.
🔥 Risk
Remote Code Execution (RCE) via uploaded PHP webshells, leading to complete server compromise.
🔧 Fix 1 — Apache: Disable PHP in uploads dir
/var/www/html/wp-content/uploads/.htaccess
<FilesMatch "\.(php|php3|php4|php5|php7|phtml|pl|py|jsp|asp|sh|cgi)$">
    Order deny,allow
    Deny from all
</FilesMatch>

<IfModule mod_php.c>
    php_flag engine off
</IfModule>
🔧 Fix 2 — Restrict MIME types (functions.php)
functions.php
add_filter( 'upload_mimes', function( $mimes ) {
    return [
        'jpg|jpeg|jpe' => 'image/jpeg',
        'gif'          => 'image/gif',
        'png'          => 'image/png',
        'webp'         => 'image/webp',
        'pdf'          => 'application/pdf',
        'doc|docx'     => 'application/msword',
        'mp4|m4v'      => 'video/mp4',
    ];
}, 1 );
👥 Responsible
Web Server Admin WordPress Developer
✅ Verification
$ curl -I https://your-site.com/wp-content/uploads/evil.php → HTTP/2 403
📋 VAPT Closure Statement
🔒
Audit Closure
PHP/script execution disabled in uploads directory via server-level config (HTTP 403). Allowed MIME types restricted at application layer to defined whitelist. Both server and app-layer controls active.

Web Application Firewall (WAF) Configuration
HIGHServer + FirewallModSecurityWordfenceNginx
⌄
⚠️ Issue
Without a WAF, the application is exposed to OWASP Top 10 attacks: SQL injection, XSS, LFI, path traversal, CSRF, and automated exploit scanners.
🔥 Risk
Exploitation of application-layer vulnerabilities in WordPress core, themes, or plugins resulting in data breach or RCE.
🔧 Fix 1 — ModSecurity + OWASP CRS
Shell
$ apt-get install libapache2-mod-security2 -y
$ a2enmod security2
$ sed -i 's/SecRuleEngine DetectionOnly/SecRuleEngine On/' /etc/modsecurity/modsecurity.conf
$ systemctl restart apache2
🔧 Fix 2 — Nginx: Manual WAF rules
/etc/nginx/sites-enabled/your-site.conf
# Block SQL injection
set $block_sqli 0;
if ( $query_string ~* "union.*select.*\(" ) { set $block_sqli 1; }
if ( $block_sqli = 1 ) { return 403; }

# Block XSS
set $block_xss 0;
if ( $query_string ~* "<script>" )    { set $block_xss 1; }
if ( $query_string ~* "javascript:" ) { set $block_xss 1; }
if ( $block_xss = 1 )                 { return 403; }

# Block known scanners
if ( $http_user_agent ~* (nikto|sqlmap|nmap|masscan|zgrab) ) { return 403; }
if ( $http_user_agent = "" ) { return 403; }
👥 Responsible
Linux/System Admin Web Server Admin
✅ Verification
$ curl -I "https://your-site.com/?id=1+UNION+SELECT+1,2,3" → HTTP/2 403
📋 VAPT Closure Statement
🔒
Audit Closure
WAF deployed at server layer (ModSecurity + OWASP CRS v4 in enforcement mode) and WordPress layer (Wordfence). SQLi, XSS, path traversal patterns actively blocked. Verified via controlled injection test payloads returning HTTP 403.

Security Headers (HTTP Response Headers)
HIGHServer LevelApacheNginxHSTSCSP
⌄
⚠️ Issue
Missing HTTP security headers leave the site vulnerable to clickjacking, MIME-sniffing, XSS via content injection, and SSL stripping.
🔥 Risk
XSS via content injection, clickjacking, session hijacking via HTTP downgrade, MIME confusion attacks.
🔧 Fix — Nginx
/etc/nginx/sites-enabled/your-site.conf
add_header X-Frame-Options "SAMEORIGIN" always;
add_header X-Content-Type-Options "nosniff" always;
add_header X-XSS-Protection "1; mode=block" always;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "geolocation=(), microphone=(), camera=(), payment=()" always;
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; img-src 'self' data: https:; font-src 'self'; frame-ancestors 'self'; base-uri 'self'; form-action 'self';" always;
server_tokens off;
fastcgi_hide_header X-Powered-By;
👥 Responsible
Web Server Admin Linux/System Admin
✅ Verification
$ curl -I https://your-site.com/ 2>&1 | grep -E '(Strict|X-Frame|X-Content|Content-Security)' → All headers present # securityheaders.com → Grade: A
📋 VAPT Closure Statement
🔒
Audit Closure
All recommended HTTP security headers implemented: HSTS (1 year + includeSubDomains), X-Frame-Options SAMEORIGIN, X-Content-Type-Options nosniff, CSP, Referrer-Policy, Permissions-Policy. Server tokens suppressed. Grade A on securityheaders.com.

14
Backup Security Essentials
MEDIUMServer + PluginUpdraftPlusmysqldumpgpg
⌄
⚠️ Issue
Backups stored in web root are publicly downloadable. Unencrypted backups expose database credentials. Missing backups prevent recovery from ransomware.
🔥 Risk
Exposed backup files leak full database dumps. No backup = no recovery. Unencrypted backups violate DPDP Act / IT Act compliance.
🔧 Fix — Automated encrypted backup script
/usr/local/bin/wp-backup.sh
#!/bin/bash
SITE_DIR="/var/www/html"
BACKUP_DIR="/var/backups/wordpress"   # OUTSIDE web root
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="wordpress_db"
DB_USER="wp_user"
DB_PASS="your_db_password"
GPG_RECIPIENT="backup@yourdomain.com"
RETENTION_DAYS=30

mkdir -p $BACKUP_DIR

# Database backup (encrypted)
mysqldump -u $DB_USER -p$DB_PASS $DB_NAME | gzip | \
    gpg --recipient $GPG_RECIPIENT --encrypt \
    > $BACKUP_DIR/db_${DATE}.sql.gz.gpg

# Files backup (encrypted)
tar -czf - --exclude="$SITE_DIR/wp-content/cache" $SITE_DIR | \
    gpg --recipient $GPG_RECIPIENT --encrypt \
    > $BACKUP_DIR/files_${DATE}.tar.gz.gpg

chmod 600 $BACKUP_DIR/*.gpg
find $BACKUP_DIR -name "*.gpg" -mtime +$RETENTION_DAYS -delete
echo "Backup completed: $DATE" >> /var/log/wp-backup.log
👥 Responsible
Linux/System Admin WordPress Developer
📋 Checklist
  • Backups stored outside web root
  • Encrypted with GPG
  • Remote/offsite copy (3-2-1 rule)
  • Permissions 600 (root only)
  • 30+ day retention
  • Restore tested quarterly
✅ Verification
$ ls -la /var/backups/wordpress/ → -rw------- 1 root root ... db_20250421_020001.sql.gz.gpg $ curl -I https://your-site.com/../backups/wordpress/ → HTTP/2 403 or 404
📋 VAPT Closure Statement
🔒
Audit Closure
GPG-encrypted daily backups stored outside web root with 600 permissions. Remote replication active (3-2-1 rule). 30-day retention enforced. Backups inaccessible via HTTP. Restore tested and documented.
15
Lock Down /wp-includes/ & /wp-content/uploads/ Public Access
CRITICAL Server + PHP Runtime ApacheNginx.htaccessPHP-FPMopen_basedir
⌄
⚠️ Issue
Both /wp-includes/ and /wp-content/uploads/ are publicly accessible with no PHP execution restrictions. wp-includes/ houses WordPress core internals that should never be directly reachable by the public. uploads/ is the most common webshell drop zone — one malicious upload + PHP execution = full RCE. Additionally, cgi.fix_pathinfo=1 (PHP default) lets attackers append /evil.php to any uploaded image URL to trigger PHP execution.
🔥 Risk
Remote Code Execution (RCE) via uploaded webshells in uploads/. Direct exploitation of WordPress core via wp-includes/. cgi.fix_pathinfo path-traversal RCE. Full server takeover, data exfiltration, ransomware, lateral movement.

🔧 Fix B2 — Nginx: Lock down /wp-content/uploads/
/etc/nginx/sites-enabled/your-site.conf — add inside server{} block
# ─── wp-content/uploads/ lockdown ────────────────────────────
location /wp-content/uploads/ {
    autoindex off;

    # HIGH PRIORITY — deny all PHP/script files. Catches double-extension tricks too.
    location ~* /wp-content/uploads/.*\.(php|php3|php4|php5|php7|php8|phtml|phar|pl|py|cgi|asp|sh|bash|rb|lua)$ {
        deny all;
        return 403;
        access_log /var/log/nginx/uploads_blocked.log;   # alert in SIEM
    }

    # Prevent SVG and HTML from executing JS (XSS vector)
    location ~* /wp-content/uploads/.*\.(svg|svgz|html|htm|xml)$ {
        add_header Content-Type text/plain;
        add_header Content-Disposition attachment;
        add_header X-Content-Type-Options "nosniff" always;
    }

    # Block hidden/dotfiles (.htaccess, .env, etc.)
    location ~* /wp-content/uploads/\. {
        deny all;
        return 403;
    }

    # Force nosniff + restrictive CSP on all uploads responses
    add_header X-Content-Type-Options "nosniff" always;
    add_header Content-Security-Policy "default-src 'none'" always;

    # Allow only safe static file types
    location ~* \.(jpg|jpeg|png|gif|webp|ico|pdf|mp4|mp3|mov|doc|docx|xls|xlsx|ppt|pptx|zip|txt|woff|woff2|ttf|eot)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
        add_header X-Content-Type-Options "nosniff" always;
    }
}

🔧 Fix C — PHP-FPM: cgi.fix_pathinfo + open_basedir + dangerous functions
/etc/php/8.2/fpm/php.ini — critical runtime hardening
; ─── CRITICAL: Prevent path-info RCE ────────────────────────────
